## 1. 서비스 예시 (상세)

### 노래방 서비스 (이상적인 시나리오)

브라우저 확장 프로그램이 사용자 환경에 UI를 주입할 수 있다는 것을 보여주는 예시.

```
유튜브 뮤직비디오 시청 시 자동 감지
    → 음원/가사 추출
    → 음원분리
    → 싱크맞추기
    → 유튜브 영상 위에 가사 UI 표시 + 보컬제거 MR 재생
```

**관련 MCP 서버:**
- 음원분리 (Demucs/Roformer 기반) - 구축 완료
- Whisper 기반 싱크 조절 - 구축 완료

### 유튜브 스폰서 광고 스킵 (MCP sampling 활용)

MCP의 sampling 기능을 활용한 예시. 서버가 클라이언트 LLM에게 completion을 요청하는 패턴.

```
유튜브 페이지 진입
    → 자막 추출
    → MCP 서버가 자막 구간 판독을 LLM 요청 (sampling)
    → 해당 구간 응답
    → 자막 구간 진입 시 자동 건너뛰기
```

**참고:** SponsorBlock과 유사한 기능이나, LLM 기반 자동 판단이 차별점.

## 2. 제거된 철학적 개념

### Fractal Holarchy

원래 프로젝트명 영감의 원천이었으나, 실제 구현과의 연관성이 불명확하여 제거됨.

**개념:**
- 부분과 전체가 유기적으로 연결
- 각 컴포넌트가 동일한 구조를 가짐
- 하나의 작동 방식을 알면 전체를 이해

**소프트웨어 적용 예시:**
- React/Angular 컴포넌트 트리
- Elm Architecture의 프랙탈 패턴

**제거 이유:**
- 구현에 직접적 영향을 주지 않음
- 프로젝트 설명을 복잡하게 만듦
- 아키텍처 결정에 실질적 도움이 되지 않음

## 3. 초기 비즈니스 모델 상세

### 소비자 구분별 제공서비스 (초기 구상)

#### 1. 공통 (개발자 대상, 일반사용자는 선택적 사용)
- MCP 서버, A2A agents 추가, 제거, 수정
- 브라우저(Chrome extension 등을 통한 프롬프트 JS 주입)에서 프롬프트 창을 통해 언제든 호출 가능
- 기본탑재 FHLY MCP 서버 활용 (예: yt-dlp 라이브러리 기반 MCP 서버) - 현재 미확정
- 구독기반(유료) FHLY MCP 서버 활용 (Demucs/roformer 음원분리, whisper 싱크조절, OCR 등)

#### 2. 일반사용자
- 구독한 MCP(또는 A2A) 서비스 이용
- 현재 개발된 MCP 서버: 음원분리, 범용(whisper) 및 특정사이트 캡차 인식 모델
- 브라우저 프롬프트로 호출하거나 자동 호출되어 작업 진행

**문제점 분석:**
- 개발자와 일반 사용자의 니즈가 완전히 다름
- 두 타겟을 동시에 만족시키기 어려움
- 우선순위: 통합 호스트 개발 → 이후 비즈니스 모델 구체화

## 4. 브라우저 통합 방식 비교

| 방식 | 채택 여부 | 이유 |
|------|-----------|------|
| MITM Proxy | 제거 | 보안 위험, 인증서 조작 필요 |
| Chrome Extension | 채택 | 표준적, 안전, 정책 준수 가능 |
| JS 주입 (확장 프로그램 통해) | 채택 | Content Script로 구현 |

## 5. 참고 자료

### 법적 고려사항
- yt-dlp: 도구 자체는 합법, 사용은 사용자 책임
- 광고 차단: EU 법원 합법 판결
- SponsorBlock: 사용자 커뮤니티 기반, 합법으로 간주
- YouTube ToS 위반 ≠ 불법 (계정 정지 위험은 있음)

### GDPR 적용
- 로컬 처리만: 해당 안 됨
- 서버 전송 시: 명시적 동의, 삭제권, 데이터 최소화 필요
